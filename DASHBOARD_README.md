# RAG Benchmark Dashboard - Usage Guide

## Quick Start

1. **Open the Dashboard**:
   ```bash
   # Windows
   start dashboard.html

   # macOS
   open dashboard.html

   # Linux
   xdg-open dashboard.html
   ```

2. **Explore the Results**:
   - Summary cards show winners in each category
   - Charts visualize trade-offs between architectures
   - Detailed table shows all metrics

---

## What Each Chart Shows

### Recall@5 Comparison
- **What**: How many relevant documents (out of all relevant docs) were in top 5?
- **Winner**: Hybrid (1.80)
- **Why it matters**: Higher recall = more relevant info retrieved

### P50 Latency
- **What**: Median response time (milliseconds)
- **Winner**: Graph (9.4ms)
- **Why it matters**: Lower latency = faster user experience

### Throughput (queries/sec)
- **What**: How many queries per second can the system handle
- **Winner**: Graph (127.8 q/s)
- **Why it matters**: Higher throughput = better for high-volume traffic

### Recall vs Latency Trade-off
- **What**: Scatter plot showing the speed/accuracy balance
- **Sweet Spot**: Vector (middle - good recall, fast latency)
- **Extremes**:
  - Upper left = high recall, slow (Hybrid)
  - Lower right = low recall, fast (Graph)

## How to Choose an Architecture

### I want Maximum Accuracy
→ **Use HYBRID RAG**
- Recall: 1.80 (20% higher than Vector)
- Latency: 188ms (still acceptable)
- Cost: Same as Vector

### I want Production-Ready (Recommended)
→ **Use VECTOR RAG**
- Recall: 1.50 (good)
- Latency: 75ms (fast)
- Complexity: Lowest
- This is your default choice

### I need Sub-10ms Latency
→ **Use GRAPH RAG**
- Latency: 9ms (100× faster than Vector!)
- Recall: 0.80 (acceptable if willing to trade)
- Limitation: Entity resolution issues

### I have Very Large Documents (>10K tokens)
→ **Use PARENT-CHILD RAG**
- Benefits from hierarchical chunking
- More RAM required (3.6 GB vs 1.6 GB)
- 4.5+ minute indexing time

### I need Query Variance Reduction
→ **Use MULTI-QUERY RAG**
- Makes 3 sub-queries per question
- 9× slower with no recall benefit
- Only if specifically tuning for consistency

## Interpreting the Metrics

| Metric | Good Value | Explanation |
|--------|-----------|------------|
| Recall@5 | > 1.0 | Out of 5 retrieved, how many are relevant (averaged across queries) |
| Precision@5 | > 0.2 | What % of retrieved documents are actually relevant |
| MRR | > 0.5 | How high ranked is the first relevant document (1/rank) |
| P50 Latency | < 200ms | 50% of queries respond in this time (user acceptable) |
| Throughput | > 5 q/s | Queries per second the system can handle |
| Peak RAM | < 4GB | Maximum memory needed during indexing/query |
| Storage | < 2MB | Index size on disk |

---

## Important Caveats

1. **These are synthetic queries** - Generated by RAGAS, not real user queries
2. **Small dataset** - 50 Wikipedia articles, may not scale to 100K+ docs
3. **Single LLM** - Uses Groq 8B; GPT-4o/Claude-3 might have different results
4. **Entity resolution issues** - Kubernetes "hub node" problem affects Graph RAG
5. **No quality metrics** - Focused on retrieval only (not hallucination/accuracy)
